# 1️⃣ 데이터 타입

### 📘 01. 데이터 타입의 종류

* 자바스크립트의 타입은 크게 <mark style="color:blue;">`기본형`</mark>과 <mark style="color:blue;">`참조형`</mark> 두 종류로 나뉜다.
* 구분 기준으로는 기본형은 기본형은 값이 담긴 <mark style="color:blue;">`주소값을 바로 복제`</mark>하고,\
  참조형은 값이 담긴 <mark style="color:blue;">`주소값으로 이뤄진 묶음을 가리키는 주소값을 복제`</mark>한다.

#### ☝️ 기본형 (원시형, Primitive type)

* 숫자(`number`), 문자열(`string`), 불리언(`boolean`), null, undefined, [심볼(`Symbol`)](#user-content-fn-1)[^1]
* 불변성(immutability)을 가짐

#### ✌️ 참조형 (Reference type)

* 객체(`object`)
* 배열(`Array`), 함수(`Function`), 날짜(`Date`), 정규표현식(`RegExp`), [Map, WeakMap, Set, WeakSet](#user-content-fn-2)[^2]이 객체의 하위 분류로 이뤄짐

### 📘 02. 데이터 타입에 관한 배경지식

#### 1️⃣ 비트(bit)와 메모리

* 컴퓨터는 모든 데이터를 0 또는 1로 저장함
* 0 또는 1로 표현할 수 있는 하나의 메모리 조각 → bit
* 비트는 고유한 위치(식별자)를 가짐

#### 2️⃣ 바이트의 등장

* 1바이트 = 8비트 (1바이트 → 2⁸ = **256개 표현 가능**)
*   비트를 묶으면 표현 범위, 검색 효율이 증가하지만 메모리 낭비가 있을 수 있음

    **(표현력 vs 메모리 효율의 트레이드 오프)**

{% hint style="info" %}
### &#x20;언어별 차이

**정적타입 언어**

* C/C++, Java는 메모리 낭비 최소화를 위해 타입별 메모리 크기를 고정함
* 메모리 표현 범위 초과 시 오류 발생
* 개발자가 직접 타입을 변경해야 하는 불편함이 있음 (`short → int`)

`→ 메모리 용량이 부족해서 위와 같은 합리적인 선택을 함`&#x20;



**자바스크립트는?**

* 자바스크립트는 메모리 용량이 커진 상황에서 등장한 언어
* 메모리 효율보다는 개발 편의성에 중점을 둠
* 정수/실수 구분 없이 숫자를 64비트로 저장함


{% endhint %}

#### 3️⃣ 메모리 주소 (<mark style="color:blue;">`= 바이트 단위의 식별자`</mark>)

* 모든 데이터는 바이트 단위의 주소값을 가짐
* 변수는 값을 직접 저장하는게 아니라 메모리 주소를 통해 값을 저장하거나 참조함

{% hint style="info" %}
### &#x20;식별자와 변수

**변수 (variable)**

* 변할 수 있는 수 `→ 변할 수 있는 무언가(데이터)`&#x20;



**식별자(identifier)**

* 어떤 데이터를 식별하는데 사용하는 이름 `→ 변수명`


{% endhint %}

### 📘 03. 변수 선언과 데이터 할당

#### 1️⃣ 변수 선언

```js
var a;
```

* 메모리에서 비어있는 공간을 확보하고, 그 공간에 식별자를 연결하는 것

#### 📌 메모리 상태

| 변수 영역 주소 | 식별자 | 값         |
| -------- | --- | --------- |
| 1003     | a   | undefined |

* 변수 공간이 하나 확보된 상태
* 아직 데이터는 없음 (`undefined`)

→ 선언은 공간 확보 + 이름(식별자) 부여

#### 2️⃣ 데이터 할당

```js
var a = 'abc';
```

* [변수 영역](#user-content-fn-3)[^3]에 a 공간 확보
* 데이터 영역 `'abc'` 저장
* 변수 a에 `'abc'`가 저장된 주소를 기록

#### 📌 메모리 상태

**🔹 변수 영역**

| 변수 영역 주소 | 식별자 | 값(주소) |
| -------- | --- | ----- |
| 1003     | a   | 5004  |

**🔹 데이터 영역**

| 데이터 영역 주소 | 값     |
| --------- | ----- |
| 5004      | 'abc' |

→ 변수에는 직접 값이 저장되는 것이 아니라 값을 저장한 `주소`를 저장

#### 3️⃣ 데이터 재할당 (값 변경)

```js
a = 'abcdef'
```

* 변수 a에 `'abcdef'`를 재할당 하면 `'abc'`에 값을 추가하는게 아니라 `'abcdef'`를 새로 생성함
* 즉, 데이터 영역의 새로운 공간에 값을 새로 저장
* 이후 a가 가리키는 주소를 변경함

#### 📌 메모리 상태

**🔹 변수 영역**

| 변수 영역 주소 | 식별자 | 값(주소) |
| -------- | --- | ----- |
| 1003     | a   | 5005  |

**🔹 데이터 영역**

| 데이터 영역 주소 | 값        |
| --------- | -------- |
| 5004      | 'abc'    |
| 5005      | 'abcdef' |

→ 기존 `'abc'`를 수정하지 않고 새로 생성

→ `'abc'`에 대한 모든 참조가 끊기면 가비지 컬렉터의 수거 대상이 됨&#x20;

#### 4️⃣ 왜 이렇게 설계했을까?

#### 이유 1) 가변 길이 데이터 처리를 위해

* 문자열은 길이가 계속 변할 수 있음
* 기존 공간을 늘린다면, 뒤에 공간이 없다면 재배치하거나 다른 데이터를 이동해줘야 함 → 비효율
*   그래서 변수 영역과 데이터 영역을 분리함

    → 값은 새로 만들고, 주소만 계속 바꾸는 구조

#### 이유 2) 중복 데이터 처리 효율

```js
var a = 5;
var b = 5;
var c = 5;
```

#### 📌 메모리 상태

**🔹 변수 영역**

| 주소   | 식별자 | 값(주소) |
| ---- | --- | ----- |
| 1001 | a   | 5001  |
| 1002 | b   | 5001  |
| 1003 | c   | 5001  |

**🔹 데이터 영역**

| 주소   | 값 |
| ---- | - |
| 5001 | 5 |

→ 5를 한 번만 저장하고 변수들은 값의 주소만 공유하면 됨 → <mark style="color:blue;">`데이터 영역 메모리 효율 증가`</mark>

→ 만약 각각 등록한다면 변수영역 3건, 데이터 영역 3건의 대한 메모리가 필요함
